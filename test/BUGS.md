# Test-suite-discovered bugs

Bugs the new `test/` suite has surfaced while running. Each entry is a
single, reproducible issue; once fixed in `src/`, remove the TODO[BUG]
comment in the corresponding test and delete the entry here.

Per project policy (CLAUDE.md), the agent does NOT touch `src/` when
finding bugs — only documents them and marks the test so the suite
stays green.

## SQLite: `ln(x)` emits `log(x)` which is base-10 in SQLite

- File: [src/sqlBuilders/SqliteSqlBuilder.ts](../src/sqlBuilders/SqliteSqlBuilder.ts) — `_ln` override (line ~281).
- Symptom: `INumberValueSource.ln()` on the sqlite connection emits
  `log(x)`. In SQLite, single-argument `log(x)` is the base-10
  logarithm, not the natural log — so `ln(exp(2))` returns ≈ `0.868`
  (i.e. `log10(e²)`) instead of `2`.
- Marked in test: `test/db/sqlite/*/select.numeric-ops.test.ts`,
  `test('exp-and-ln', ...)`. The test still asserts SQL / params /
  type so future SQL-emission regressions are caught; the real-DB
  data assertion is skipped on sqlite until the fix lands.
- Suggested fix: emit `ln(x)` (SQLite >= 3.35 ships a `ln()` extension)
  or `log(?, x)` with an explicit base (`log(2.718281828..., x)` is
  fragile; better to require the math extension and call `ln(x)`).

## Oracle: docs mention `UPDATE … FROM …` but the type rejects it

- File: [src/expressions/update.ts](../src/expressions/update.ts)
- Symptom: [docs/queries/update.md](../docs/queries/update.md) shows
  Oracle SQL for `update(table).from(other)`, but the
  `UpdateExpression.from` signature lists only
  `'noopDB' | 'mariaDB' | 'mySql' | 'postgreSql' | 'sqlite' | 'sqlServer'`
  in its database constraint — Oracle is excluded.
- Marked in test: [test/db/oracle/newest/oracledb/update.from.test.ts](db/oracle/newest/oracledb/update.from.test.ts)
  — the test is commented out with a TODO[BUG] note so the symmetry
  audit still finds the same file in every cell.
- Suggested fix: either align the type signature with the docs (add
  `'oracle'` to the constraint and add the dialect's SQL emission) or
  rewrite the docs to acknowledge the limitation.

## extractWritableColumnsFrom: type vs runtime inconsistency

- Files: [src/Table.ts](../src/Table.ts) (column() returns `Column`,
  which extends `WritableDBColumnWithoutDefaultValue` → therefore
  `WritableDBColumn`), and
  [src/extras/utils.ts](../src/extras/utils.ts) (the runtime
  filters by `columnPrivate.__writable`).
- Symptom: `WritableColumnKeys<typeof t>` includes every column
  declared with `this.column(...)`, but `extractWritableColumnsFrom`
  returns a strict subset: only `optionalColumn`,
  `columnWithDefaultValue`, and `autogeneratedPrimaryKey` columns.
  Required, no-default columns (`status`, `priority`, etc.) are
  silently dropped at runtime.
- Marked in test:
  [test/db/*/docs.advanced.columns-from-object.test.ts](db/sqlite/newest/bun_sqlite/docs.advanced.columns-from-object.test.ts),
  `extract-writable-columns` — asserts the present runtime behaviour
  with a TODO[BUG] note.
- Suggested fix: either set `__writable = true` on `this.column()`
  at construction time so the runtime matches the type, or narrow
  the type to only include the column kinds the runtime accepts.
